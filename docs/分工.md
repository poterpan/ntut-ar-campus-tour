## 策略

Unity 場景的協作衝突是團隊開發最痛的點,因此本專案**把所有「會動主 Unity 場景的事」集中由 A 主導**(透過 AI agent 處理),其他三人做「交付獨立成品給 A 接」的工作。三人的成品互不依賴、可完全並行。

不追求工作量平均,以**快速迭代、能整合出產品**為主。

## 模組介面約定

W1 必須定案 —— 介面講清楚,平行開發才不會卡。

### POI 資料結構

主場景以下面這個 schema 載入 POI(JSON 或 ScriptableObject 都行):

```json
{
  "id": "p01",
  "name": "新生南路側門",
  "lat": 25.041939,
  "lng": 121.535678,
  "anchorType": "Terrain",
  "shortDescription": "(C 填:30 秒內可念完的開場介紹,給 TTS 用)",
  "llmSystemPrompt": "(C 填:壓縮自 docs/POI.md,作為 LLM 對話的 system prompt 上下文)"
}
```

- 座標、長版解說內容、anchorType:見 `docs/POI.md`
- `shortDescription` / `llmSystemPrompt`:由 C 組員 W1 內回填到 POI.md(可於每節下追加「短版」子段)

### NPC ↔ LLM 介面

B 組員實作模組對外提供:

```csharp
public interface ILlmClient {
    // 輸入問題 + 當下 POI 上下文,回傳一段純文字回答(供 TTS 念出)
    Task<string> AskAsync(string question, PoiContext poi, CancellationToken ct = default);
}

public record PoiContext(string Id, string Name, string LlmSystemPrompt);
```

B 端職責:API 金鑰管理、模型選擇(Claude / GPT / Gemini)、prompt 串接、錯誤處理(network 斷線降級)。

### 校園貓 ONNX 規格

D 組員照此 spec 設計 Agent,訓練好的 ONNX 給 A:

**觀察 input(`[1, 7]` float,順序固定):**

| # | 欄位 | 計算方式 | 正規化 |
|---|---|---|---|
| 0–1 | `food_local_x`, `food_local_z` | 最近飼料相對貓的 local 座標 | / MAX_RANGE(5m) |
| 2 | `food_dist` | sqrt(x²+z²) | / MAX_RANGE |
| 3 | `food_exists` | 1 有 / 0 無 | flag |
| 4–5 | `vel_local_x`, `vel_local_z` | 自身速度(local frame) | / MAX_SPEED(0.5 m/s) |
| 6 | `hunger` | 飢餓值,隨時間遞增 | 已是 [0, 1] |

**動作 output(`[1, 2]` float 連續,值域 [-1, 1]):**

| # | 欄位 | Unity 端換算 |
|---|---|---|
| 0 | `forward_speed` | `vel = action[0] × MAX_SPEED` |
| 1 | `turn_rate` | `rot += action[1] × MAX_TURN × dt`(MAX_TURN = 90°/s) |

訓練吃不消可改 discrete(branch size = 4:idle / forward / turn-left / turn-right),但 D 要先告知 A,A 再調整推論端的 action 解碼。詳細 reward、訓練環境、觸發機制見 `docs/貓互動設計.md`。
